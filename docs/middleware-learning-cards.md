# 中间件使用场景与面试复盘卡片

本文只基于当前仓库已经实现、测试和记录过的内容整理。它用于复习和面试准备，不把本地 Docker Compose Demo 包装成生产经验。

## 复习方法

每个中间件按同一套问题复盘：

1. 为什么要引入它。
2. 它在本项目解决了什么业务问题。
3. 当前 Demo 实际用了哪些能力。
4. 企业开发中还会怎么用。
5. 当前 Demo 和生产系统有什么差距。
6. 面试中怎么回答，不夸大、不编造。

## Redis

### 为什么使用

Redis 用来缓解高频读请求直接打到数据库的问题。当前 Demo 选择商品查询作为场景，因为商品、科室、医生这类读多写少数据很适合先从缓存开始学习。

### 本项目中的业务问题

`redis-cache-demo` 实现商品查询缓存旁路模式：

- 查询商品时先读 Redis。
- Redis 未命中再查询模拟数据库。
- 查到商品后回填 Redis，并设置 TTL。
- 更新商品后删除缓存，避免继续读到旧值。
- 对不存在商品写短 TTL 空值，降低缓存穿透。
- 对热点 key 回源加本地互斥，降低缓存击穿。
- 对正常缓存 TTL 增加随机抖动，降低同一时间大批 key 过期的雪崩风险。

### 当前 Demo 用到的能力

- Spring Data Redis 的 `StringRedisTemplate`。
- 缓存旁路、TTL、更新后删除缓存。
- 空值缓存：`__MISSING__`。
- 单 JVM 本地互斥锁。
- TTL 随机抖动。
- 单元测试和本地 Redis HTTP 联调。

### 企业级常见用法

- 热点数据缓存：商品详情、医生信息、科室列表、配置快照。
- 缓存一致性策略：更新数据库后删除缓存，必要时延迟双删或订阅变更事件。
- 缓存穿透治理：短 TTL 空值、布隆过滤器、参数校验。
- 缓存击穿治理：互斥锁、逻辑过期、热点 key 预热、异步刷新。
- 缓存雪崩治理：TTL 抖动、多级缓存、限流降级、容量预案。
- 分布式锁：控制跨实例互斥，但要理解过期时间、续期、锁误删和 Redisson 的边界。
- 高可用部署：主从、哨兵、Cluster，配合持久化和监控。

### Demo 与生产差距

- 当前 repository 是内存 Map，不是 MySQL。
- 本地互斥锁只保护单 JVM，多实例需要分布式锁、逻辑过期或异步刷新。
- Redis 是本地单容器，不证明主从、哨兵或 Cluster 高可用。
- 没有压测，不能写吞吐量提升、P95/P99 或容量结论。

### 面试问答

**Q1：什么是缓存旁路模式？**  
A：应用先查缓存，缓存未命中再查数据库，查到后写回缓存。更新时通常先更新数据库，再删除缓存。当前 Demo 的 `ProductService` 就是这个流程。

**Q2：为什么更新后删除缓存，而不是直接更新缓存？**  
A：删除缓存更简单，能让下一次查询重新从数据库加载新值；直接更新缓存容易遗漏复杂字段、并发顺序和失败回滚问题。

**Q3：缓存穿透是什么，当前怎么处理？**  
A：大量请求查询不存在的数据，缓存和数据库都没有结果，导致请求不断打到数据库。当前 Demo 对不存在商品写入短 TTL 的 `__MISSING__`。

**Q4：缓存击穿是什么，当前怎么处理？**  
A：热点 key 过期瞬间，大量请求同时回源数据库。当前 Demo 对同一个商品 ID 使用本地互斥锁，并在锁内二次检查缓存。

**Q5：缓存雪崩是什么，当前怎么处理？**  
A：大量 key 同时过期或 Redis 故障，导致请求集中打到数据库。当前 Demo 只做了 TTL 随机抖动，降低同一时间过期概率。

**Q6：Redis 持久化 RDB 和 AOF 怎么理解？**  
A：RDB 是定期生成快照，恢复快但可能丢最近一段数据；AOF 记录写命令，数据更完整但文件更大、恢复更慢。生产中常结合使用。

**Q7：主从、哨兵、Cluster 分别解决什么？**  
A：主从解决读扩展和副本备份；哨兵解决主节点故障自动切换；Cluster 解决分片扩容和更大数据量。

**Q8：能不能说当前项目实现了 Redis 高可用？**  
A：不能。当前只验证单容器 Redis 的缓存模式和风险缓解，不证明生产高可用。

## RocketMQ

### 为什么使用

RocketMQ 用来把预约创建和通知发送解耦。HTTP 请求只负责保存预约并发布事件，耗时、可重试的通知发送交给消费者异步处理。

### 本项目中的业务问题

`rocketmq-notification-demo` 实现医疗预约创建后的异步通知：

- 创建预约后发布 `AppointmentCreatedEvent`。
- Producer 使用 `topic:tag` 发送消息。
- Consumer 消费预约事件并发送通知。
- 使用 `eventId` 做幂等。
- 消费失败时抛异常，交给 RocketMQ 重试。
- 用 `patientId=500` 做稳定失败重试演练。

### 当前 Demo 用到的能力

- RocketMQ Spring Boot Starter。
- `RocketMQTemplate.syncSend`。
- `@RocketMQMessageListener`。
- topic、tag、message key。
- 消费幂等和失败重试。
- 单 NameServer、单 Broker Docker Compose 联调。

### 企业级常见用法

- 异步解耦：订单、预约、支付后通知、积分发放。
- 削峰填谷：把高峰请求写入 MQ，由消费者按能力处理。
- 可靠消费：幂等表、状态机、重试、死信队列和告警。
- 顺序消息：同一订单或同一用户维度保证局部顺序。
- 延迟消息：预约提醒、订单超时取消。
- 事务消息：本地事务和消息发送需要最终一致时使用。
- 消息堆积治理：扩容消费者、排查慢消费、限流生产者、拆分 topic。

### Demo 与生产差距

- 当前是本地单 NameServer、单 Broker，不证明生产高可用。
- 没有压测吞吐、积压恢复耗时或 broker 级监控。
- 幂等只证明通知记录不重复写，不是精确一次消费。
- 没有演示顺序消息、延迟消息、事务消息和死信队列。

### 面试问答

**Q1：为什么不在创建预约接口里直接发送通知？**  
A：通知可能慢、失败或依赖外部服务。放进 MQ 后，预约创建链路更短，通知失败可以由消费者重试和补偿。

**Q2：Producer 同步发送成功代表什么？**  
A：代表消息成功发送到 Broker，不代表消费者已经处理，也不代表业务最终成功。

**Q3：为什么 Consumer 必须幂等？**  
A：MQ 可能因为重试、网络抖动、消费者重启导致重复投递。业务必须能承受重复消息。

**Q4：当前项目如何做幂等？**  
A：通知记录落 MySQL，`notification_records.event_id` 有唯一索引，重复事件只能写入一条通知记录。

**Q5：消费失败为什么要抛异常？**  
A：如果吞掉异常并返回成功，RocketMQ 会认为消息已消费，后续不会重试。抛异常才能触发重投。

**Q6：顺序消息适合什么场景？**  
A：同一业务键下有先后依赖的事件，例如同一订单的创建、支付、发货。通常按业务键选择队列。

**Q7：事务消息解决什么？**  
A：解决本地事务和消息发送之间的一致性问题，例如订单创建成功后必须最终发出订单事件。

**Q8：消息堆积怎么排查？**  
A：看生产速度、消费速度、消费者错误率、单条消息耗时、下游依赖是否变慢，再决定扩容消费者、限流或修复慢逻辑。

## MySQL

### 为什么使用

MySQL 存业务事实数据。RocketMQ 消息可能重复，幂等结果不能只放内存，必须落到有唯一约束的数据库中。

### 本项目中的业务问题

RocketMQ Stage 1 把预约和通知记录从内存 Map 改为 MySQL：

- `appointments` 保存预约主数据。
- `notification_records` 保存通知处理结果。
- `event_id` 唯一索引拦截重复消费。
- 应用重启后仍能查询已创建预约和通知记录。

### 当前 Demo 用到的能力

- MySQL 8 Docker Compose。
- Spring `JdbcTemplate`。
- 表结构初始化。
- 唯一索引幂等。
- 数据重启后保留。

### 企业级常见用法

- 业务主数据落库：用户、预约、订单、通知记录。
- 唯一索引保证幂等：请求号、事件 ID、业务流水号。
- 事务保证同一业务操作的原子性。
- 索引设计支撑查询性能。
- 慢 SQL 排查：执行计划、索引命中、回表、排序、扫描行数。
- 锁与隔离级别：处理并发更新、超卖、重复提交。

### Demo 与生产差距

- MySQL 是本地单实例 Compose，不证明主从复制、备份恢复或高可用。
- 表规模很小，没有慢 SQL 和执行计划调优证据。
- 没有复杂事务、死锁复现或大数据量索引实验。

### 面试问答

**Q1：为什么幂等要落 MySQL？**  
A：内存状态会随应用重启丢失，多实例也无法共享。数据库唯一索引是更可靠的最终防线。

**Q2：唯一索引如何保证消费幂等？**  
A：同一个 `eventId` 第一次插入成功，重复插入会被唯一索引拒绝，业务捕获后按已处理返回。

**Q3：事务隔离级别有哪些？**  
A：读未提交、读已提交、可重复读、串行化。MySQL InnoDB 默认通常是可重复读。

**Q4：索引是不是越多越好？**  
A：不是。索引能加速查询，但会增加写入成本和存储成本。要按高频查询条件和排序方式设计。

**Q5：慢 SQL 怎么排查？**  
A：先看慢日志和执行计划，确认是否全表扫描、索引失效、排序临时表、返回行数过大，再调整 SQL、索引或分页方式。

**Q6：行锁和表锁有什么区别？**  
A：行锁粒度小、并发高，但需要命中索引；表锁粒度大、并发低。InnoDB 通常使用行锁。

**Q7：当前项目能写 MySQL 调优经验吗？**  
A：不能。当前只证明落库、唯一索引幂等和重启后数据保留，没有慢 SQL 调优证据。

## Elasticsearch

### 为什么使用

Elasticsearch 用来做关键词搜索和分页检索体验。医生姓名、科室、专长这类文本检索不适合只靠数据库精确查询。

### 本项目中的业务问题

RocketMQ Stage 2 增加医生搜索：

- MySQL 存医生主数据。
- Elasticsearch 保存可重建搜索索引。
- 支持按姓名、科室、专长搜索。
- 支持分页返回。
- 删除 ES 索引后可从 MySQL 全量重建。

### 当前 Demo 用到的能力

- ES 单节点 Docker Compose。
- 创建 `doctors` 索引。
- bulk 写入医生数据。
- 搜索和分页。
- 手工重建索引。

### 企业级常见用法

- 商品、医生、文章、日志搜索。
- mapping 设计：字段类型、分词器、keyword/text 区分。
- 数据同步：DB 事件、MQ、定时任务、CDC。
- 索引重建：新索引构建完成后用别名切换。
- 深分页治理：`search_after`、scroll、业务限制最大页数。
- 索引补偿：发现丢失、漂移后从事实数据源恢复。

### Demo 与生产差距

- ES 是本地单节点，不证明集群高可用。
- 当前样例数据只有少量医生，不证明大数据量搜索性能。
- 当前主要是全量重建，没有实现 MQ 增量同步、CDC 或别名无停机切换。
- 没有自定义复杂分词和相关性调优。

### 面试问答

**Q1：为什么 MySQL 是事实数据源，ES 不是？**  
A：MySQL 保存业务真实状态和事务数据，ES 只为搜索体验服务。ES 索引可以删除后从 MySQL 重建。

**Q2：mapping 为什么重要？**  
A：mapping 决定字段类型、是否分词、如何搜索和排序。设计错了后期修改成本高，通常要重建索引。

**Q3：text 和 keyword 有什么区别？**  
A：text 会分词，适合全文搜索；keyword 不分词，适合精确匹配、聚合和排序。

**Q4：深分页有什么问题？**  
A：页数越深，ES 需要跳过和排序的数据越多，内存和 CPU 压力变大。生产中会限制页数或使用 `search_after`。

**Q5：索引重建怎么做到更平滑？**  
A：先建新索引并导入数据，校验后把别名从旧索引切到新索引，减少停机和不一致窗口。

**Q6：ES 和 MySQL 数据不一致怎么办？**  
A：先明确 MySQL 为准，再通过 MQ 重放、定时补偿、全量重建或人工修复恢复索引。

**Q7：当前项目能写 ES 调优吗？**  
A：不能。当前只证明医生搜索、分页和索引重建，没有大规模调优证据。

## Nacos

### 为什么使用

Nacos 用来学习服务注册和配置中心。当前项目先不拆成多个服务，而是在单体中验证注册、配置读取和动态刷新。

### 本项目中的业务问题

RocketMQ Stage 3 使用 Nacos：

- 应用启动后注册 `appointment-notification-service`。
- 注册元数据表达 `appointment`、`notification`、`search` 业务边界。
- 从 Nacos 读取医生搜索默认分页大小。
- 修改配置后应用无需重启即可刷新。

### 当前 Demo 用到的能力

- Nacos standalone Docker Compose。
- Nacos 原生 client。
- 服务注册。
- 配置读取和监听刷新。
- 配置影响真实搜索接口默认分页。

### 企业级常见用法

- 服务注册发现：服务间通过服务名调用，不写死 IP。
- 配置中心：开关、阈值、分页默认值、灰度参数。
- 配置灰度：按环境、分组、命名空间或实例范围发布。
- 多环境隔离：dev/test/prod 使用 namespace 或独立集群。
- 服务健康检查和实例上下线。

### Demo 与生产差距

- 当前未拆成多 Maven 服务，只是注册单体应用。
- Nacos 是 standalone，不证明集群高可用。
- 只验证一个配置项，不包含复杂灰度、权限和配置审计。
- 使用原生 client，不是完整 Spring Cloud Alibaba 微服务体系。

### 面试问答

**Q1：Nacos 注册发现解决什么问题？**  
A：解决服务实例动态变化时调用方不应写死 IP 的问题，让服务通过名称发现可用实例。

**Q2：配置中心解决什么问题？**  
A：让运行中的应用读取统一配置，并在部分场景下动态刷新，减少重新发版。

**Q3：当前项目为什么没有直接拆服务？**  
A：单体业务边界还可以讲清楚，过早拆服务会增加启动顺序、网络调用和排障成本。当前先验证治理能力。

**Q4：Nacos 配置刷新影响了哪个业务参数？**  
A：医生搜索默认分页大小 `app.search.default-page-size`。

**Q5：配置灰度是什么？**  
A：配置不一次性发给全部实例，而是按环境、分组、实例或用户范围逐步生效，降低风险。

**Q6：当前能写微服务治理经验吗？**  
A：要保守写。可以写本地服务注册和配置刷新实验，不能写生产微服务治理或 Nacos 高可用。

## Sentinel

### 为什么使用

Sentinel 用来保护接口，避免流量超过服务处理能力。当前项目选择预约创建和医生搜索两个真实业务接口，而不是做孤立限流 Demo。

### 本项目中的业务问题

RocketMQ Stage 4 接入 Sentinel：

- `POST /api/appointments` 对应 `appointment.create`。
- `GET /api/search/doctors` 对应 `doctor.search`。
- 超过本地 QPS 规则返回 HTTP 429。
- 限流命中写日志，并记录 Prometheus 指标。

### 当前 Demo 用到的能力

- Sentinel 本地 QPS 流控规则。
- `SphU.entry(resource)` 包裹业务入口。
- 限流异常转 HTTP 429。
- 真实接口限流演练。

### 企业级常见用法

- 流量控制：按接口 QPS、调用方、线程数限制。
- 熔断降级：下游慢或错误率高时快速失败。
- 热点参数限流：对某个医生、商品、用户等热点参数单独保护。
- 规则动态下发：控制台、Nacos、Apollo 等。
- 降级策略：返回缓存、兜底结果、排队或提示稍后重试。

### Demo 与生产差距

- 当前只使用本地内存 QPS 规则。
- 没有接入控制台或动态规则下发。
- 没有演示熔断、降级、热点参数限流。
- 没有压测数据，不能写高并发治理结论。

### 面试问答

**Q1：限流解决什么问题？**  
A：限制入口请求速度，避免服务或下游被瞬时流量打垮。

**Q2：为什么限流返回 429？**  
A：429 表示请求过多，语义上适合告诉调用方稍后重试。

**Q3：限流和熔断有什么区别？**  
A：限流关注入口流量是否超过阈值；熔断关注下游调用失败率、慢调用比例等是否异常。

**Q4：降级是什么？**  
A：当完整服务不可用或成本太高时，返回兜底结果，例如热门医生、缓存结果或友好提示。

**Q5：热点参数限流适合什么？**  
A：某个医生、商品、文章突然成为热点时，只限制这个热点参数，避免影响所有请求。

**Q6：当前项目 Sentinel 的边界是什么？**  
A：只验证本地规则保护真实接口，不证明控制台治理、集群限流或生产高并发。

## XXL-JOB

### 为什么使用

XXL-JOB 用来做可调度、可观测的补偿任务。当前项目用它补偿 ES 医生索引，而不是替代主链路。

### 本项目中的业务问题

RocketMQ Stage 5 增加 `doctorIndexCompensationJob`：

- 当 ES `doctors` 索引缺失或漂移时，从 MySQL 医生主数据全量重建。
- 提供 `@XxlJob` handler。
- 提供手工 HTTP 触发入口。
- 补偿逻辑和手工入口复用同一个 Service。

### 当前 Demo 用到的能力

- XXL-JOB handler 代码。
- 本地手工触发补偿接口。
- ES 删除后补偿恢复演练。
- 补偿结果记录。

### 企业级常见用法

- 失败通知补偿。
- ES 索引补偿或重建。
- 过期预约、超时订单清理。
- 分片任务处理大批量数据。
- 任务失败重试、执行日志、告警通知。
- 执行器集群和路由策略。

### Demo 与生产差距

- 默认未连接 XXL-JOB Admin。
- 没有验证 Admin 调度、失败重试、分片广播或执行器集群。
- 没有配置告警通知。
- 补偿数据量很小，不证明大批量任务性能。

### 面试问答

**Q1：为什么补偿任务不能替代主链路？**  
A：主链路要尽量实时、明确地完成业务状态变更；补偿任务用于修复异常和最终一致，不能掩盖主链路设计问题。

**Q2：当前补偿任务补偿什么？**  
A：补偿 ES 医生索引，从 MySQL 读取医生主数据并重建 `doctors` 索引。

**Q3：任务为什么要幂等？**  
A：定时任务可能失败重试或重复触发，不幂等会造成重复通知、重复写入或状态错乱。

**Q4：分片任务解决什么？**  
A：大批量数据处理时，把任务按分片分给多个执行器，提高处理能力。

**Q5：执行日志和告警为什么重要？**  
A：任务通常无人值守运行，必须能知道什么时候失败、失败原因和影响范围。

**Q6：当前能写 XXL-JOB 生产调度经验吗？**  
A：不能。当前只验证 handler 和本地手工入口，没有接入 Admin 调度和告警。

## Prometheus/Grafana

### 为什么使用

Prometheus/Grafana 用来观察系统是否正常，而不是为了放一个面板。当前项目采集 HTTP 和业务指标，服务于排障问题。

### 本项目中的业务问题

RocketMQ Stage 6 增加本地观测：

- Actuator 暴露 `/actuator/prometheus`。
- Prometheus 抓取应用指标。
- Grafana 验证本地入口可访问。
- 记录预约事件发布、通知消费、ES 补偿、Sentinel 限流等业务指标。

### 当前 Demo 用到的能力

- Spring Boot Actuator。
- Micrometer Prometheus registry。
- Prometheus Docker Compose。
- Grafana Docker Compose。
- 自定义业务 counter 和 timer。
- Prometheus target up 验证。

### 企业级常见用法

- HTTP QPS、延迟、错误率。
- MQ 消费成功、失败、积压。
- 任务执行次数、耗时、失败数。
- Sentinel 限流次数。
- Grafana Dashboard。
- Prometheus Alertmanager 告警规则。
- 和日志、链路追踪一起排障。

### Demo 与生产差距

- Grafana 只验证登录入口，没有固化 dashboard JSON。
- 没有 Alertmanager 和告警规则。
- 没有 RocketMQ broker 内部指标。
- 没有压测，所以不能写 P95/P99、容量评估或吞吐结论。

### 面试问答

**Q1：为什么要采集业务指标？**  
A：HTTP 指标只能说明接口层情况，业务指标能回答事件是否发布、通知是否消费、补偿是否成功、限流是否发生。

**Q2：Prometheus pull 模式是什么？**  
A：Prometheus 定期从目标服务的指标端点拉取数据。当前抓取 `host.docker.internal:8081/actuator/prometheus`。

**Q3：Grafana 解决什么？**  
A：把指标可视化成 Dashboard，便于观察趋势和排障。

**Q4：为什么不能写 P95/P99？**  
A：当前没有压测场景、机器规格、数据规模和持续统计，写 P95/P99 会变成无证据结论。

**Q5：`sentinel_blocks_total` 能说明什么？**  
A：说明某个 Sentinel 资源被限流的次数，可用于判断是否有流量突增或规则过严。

**Q6：补偿任务为什么要记录耗时和结果？**  
A：可以判断任务是否变慢、是否失败、是否真的完成了索引恢复。

## 一条业务链路串讲

可以这样把多个中间件串起来：

1. 用户创建预约，请求进入预约接口。
2. Sentinel 保护预约创建接口，超过 QPS 返回 429。
3. 预约主数据写入 MySQL。
4. 应用发布 RocketMQ 预约创建事件。
5. Consumer 异步发送通知，并用 MySQL `eventId` 唯一索引保证幂等。
6. 医生主数据存在 MySQL，搜索体验由 Elasticsearch 提供。
7. 如果 ES 索引缺失，XXL-JOB 补偿任务从 MySQL 重建索引。
8. Nacos 保存服务注册和搜索分页配置。
9. Prometheus 采集 HTTP、通知、补偿、限流等指标，Grafana 用于本地观察。
10. Redis 在独立 Demo 中验证缓存旁路和缓存风险处理，后续融合时可用于热点医生或科室缓存。
