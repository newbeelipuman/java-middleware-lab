# 简历驱动的 Java 中间件学习路线

## 原则：包装事实，不编造事实

简历可以强调技术决策、验证过程和结果，但每一句都应经得住追问。

建议把“吹牛”理解为：把已经完成的事情写得准确、有重点、有证据。不要写无法
复现的百万并发、性能数字、调优经验、Kubernetes 生产经验或高可用结论。

本仓库后续每个 Demo 都应留下四类证据：

1. 可运行代码。
2. 自动化测试。
3. 一键启动或复现命令。
4. 测试输出、压测结果或故障演练记录。

## 项目改造方向

现有“综合医疗服务管理系统”适合作为主项目。不要一次性把所有中间件堆进去，
应按业务场景逐步增加能力：

| 阶段 | 业务场景 | 中间件 | 可证明的简历关键词 |
| --- | --- | --- | --- |
| 1 | 医药商品或科室信息查询 | Redis | 缓存旁路、TTL、缓存失效 |
| 2 | 预约成功后异步通知 | Kafka 或 RocketMQ | 异步解耦、幂等消费、重试 |
| 3 | 药品或医生检索 | Elasticsearch | 倒排索引、分页、数据同步 |
| 4 | 多服务配置与注册 | Nacos | 注册发现、配置管理 |
| 5 | 接口保护 | Sentinel | 限流、熔断、降级 |
| 6 | 定时清理过期预约 | XXL-JOB | 调度、重试、幂等 |
| 7 | 线上定位慢接口 | Prometheus、Grafana、SkyWalking | 指标、监控、链路追踪 |

消息队列建议先选一个深入：国内 Java 业务开发可以优先 RocketMQ；若希望兼顾
数据平台、日志与通用市场覆盖，可以优先 Kafka。完成一个后再做对比实验。

## 每个技术的验收门槛

### Redis

- 实现缓存旁路、TTL 和更新失效。
- 增加缓存穿透、击穿、雪崩实验。
- 解释 RDB、AOF、主从、哨兵和 Cluster 的适用边界。
- 使用压测对比有缓存与无缓存的结果。

### Kafka 或 RocketMQ

- 实现生产者和消费者。
- 实现消费幂等与失败重试。
- 演示重复消息、消息积压和消费者重启。
- 记录吞吐量、延迟和错误率。

### Elasticsearch

- 建立商品索引与 Mapping。
- 实现关键词搜索和分页。
- 演示 MySQL 与索引的数据同步策略。
- 解释深分页和索引重建。

### 微服务治理

- 使用 Nacos 完成服务注册和配置管理。
- 使用 Sentinel 演示限流、熔断和降级。
- 使用 Gateway 完成路由和统一鉴权入口。
- 用 Docker Compose 提供最小可复现环境。

## 学习与验证流程

### 一轮学习循环

1. 先明确业务场景、技术选型、验收标准和可能的面试追问。
2. 实现最小 Demo、测试、Compose 文件和运行说明。
3. 按 README 运行项目，并逐个阅读核心类。
4. 设计一个可观察故障，例如停掉 Redis 或重复投递消息。
5. 先独立分析现象，再补充代码、测试或日志记录。
6. 运行测试和必要的联调命令，把结果写入实现日志。
7. 只有证据完整后，再整理简历表述和面试追问题库。

### 推荐任务模板

开始一个新技术时：

```text
基于医疗服务系统的预约通知场景，生成 RocketMQ 最小 Demo。
先说明业务问题、消息可靠性边界和验收标准，再实现代码。
必须包含 Docker Compose、自动化测试、故障演练和实现日志。
```

理解已有代码：

```text
逐个解释 ProductService 和 RedisProductCache。
先画出请求链路，再列出我必须独立回答的 10 个面试问题。
不要立即给答案，等我回答后再批改。
```

做故障演练：

```text
为当前 Redis Demo 设计缓存穿透、击穿和雪崩实验。
每个实验给出复现命令、预期现象、监控指标和修复方案。
实现后运行验证，并把证据写入 docs/implementation-log.md。
```

生成简历表述：

```text
只基于仓库中已验证的代码、测试和日志生成 3 条简历项目描述。
每条都要列出证据文件和可能的面试追问。
禁止使用无法证明的并发量、性能或高可用结论。
```

### 仓库约定

- `AGENTS.md`：保存长期协作规则，例如测试要求、公开仓库安全规则和简历证据规则。
- `docs/implementation-log.md`：记录每次实验和验证输出。
- 每个 Demo 的 `README.md`：保存一键复现命令和阅读顺序。
- `rfs/`：作为只读简历参考；明确要求前不修改原简历。

当某个流程稳定且会重复使用时，可以进一步整理成固定检查清单，按相同标准
实现 Demo、运行测试、记录指标和审计简历表述。

## 下一步

继续完善 Redis，而不是立刻堆叠更多中间件：

1. 缓存穿透：为空结果设置短 TTL。
2. 缓存击穿：热点 key 失效时增加互斥保护。
3. 缓存雪崩：TTL 增加随机抖动。
4. 使用压测工具记录无缓存与有缓存时的吞吐量、P95 和错误率。
5. 将结果写入实现日志，生成第一组可证明的面试回答。
